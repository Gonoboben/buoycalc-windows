# BuoyCalc Windows v0.38.1

Windows-ветка проекта BuoyCalc на C# + Avalonia.

## Статус

v0.38.1 уточняет логику вердикта после сравнения нескольких PDF-вариантов. Физику волновой нагрузки этот шаг не меняет: волна остаётся в расчёте как консервативная горизонтальная нагрузка.

## Что изменилось в v0.38.1

1. Отрицательный или нулевой вес якоря в воде теперь считается жёсткой ошибкой:

```text
ERROR: якорь имеет нулевой или отрицательный вес в воде; удержание невозможно
```

2. Такая ошибка имеет приоритет над остальными рисками и переводит вариант в:

```text
Не подходит
```

3. Для грузового/бетонного якоря на каменистом грунте логика стала мягче, если после коэффициента грунта запас якоря всё равно большой.
4. Порог большого запаса для такого случая:

```text
AnchorReserve >= 10
```

5. Если запас большой, каменистый грунт больше не даёт `WARNING`. Вместо этого добавляется инженерное примечание:

```text
INFO: каменистый грунт снижает предсказуемость работы грузового якоря, но запас якоря большой
```

6. Если это единственное замечание, вердикт становится:

```text
Подходит с примечанием
```

7. Если запас якоря на камне малый, предупреждение остаётся:

```text
WARNING: каменистый грунт может ухудшать работу грузового якоря
```

8. Волновая нагрузка не отключается и не ослабляется, даже если геометрия показывает погружённый буй. В таком случае предупреждение относится к геометрии постановки, а не к выключению волны.
9. Отображаемая версия окон и `AppInfo` обновлены:

```text
v0.38.1 - verdict cleanup
```

## Почему это важно

В одном из проверенных вариантов стоял бетонный якорь 3000 kg на камне. Расчёт давал очень большой запас якоря, поэтому итог `Требуется проверка` был слишком строгим. Камень действительно снижает предсказуемость контакта грузового якоря с дном, но при огромном запасе это должно быть примечание, а не главный отказ.

Для варианта с отрицательным весом якоря в воде наоборот нужна более жёсткая логика: такой якорь не должен считаться удерживающим элементом.

## Что осталось от v0.38

PDF-страница сравнения форм остаётся:

```text
Сравнение форм X/Z
```

Она по-прежнему показывает основную и альтернативную форму, дискретные X/Z-точки и статус альтернативной формы.

## Как обновиться локально

```text
Git -> Pull
Build -> Clean Solution
Build -> Rebuild Solution
```

## Как проверить

1. Пересчитать вариант с бетонным якорем 3000 kg на камне.
2. Если запас якоря большой, вердикт должен стать:

```text
Подходит с примечанием
```

3. В проверках должно остаться `INFO` по каменистому грунту, а не `WARNING`.
4. Пересчитать вариант с отрицательным весом якоря в воде.
5. Он должен получить:

```text
Не подходит
```

и главный риск про невозможность удержания якорем.
6. Проверить GitHub Actions:

```text
Actions -> .NET Build
```

## Следующий шаг

После проверки этих двух крайних случаев можно продолжить сравнение нескольких PDF-вариантов и уже потом переходить к итерационному solver:

```text
форма -> силы -> натяжения -> дискретные узлы -> новая форма -> сходимость
```
