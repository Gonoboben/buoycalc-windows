# BuoyCalc Windows v0.39.2

Windows-ветка проекта BuoyCalc на C# + Avalonia.

## Статус

v0.39.2 соответствует плану проекта: после каркаса итерационного solver и явных критериев сходимости расширена диагностика отчёта по итерациям.

Главный принцип остаётся прежним: старый `MooringShapeSolver` не заменяется, 2D/PDF-схемы пока не переключаются на новую форму.

## План версий

```text
v0.39   — каркас итерационного solver
v0.39.1 — критерии сходимости solver
v0.39.2 — отчёт по итерациям
v0.40   — включение дискретных нагрузок в основной solver
v0.41   — режимы постановки: surface/submerged/short/excess line
v0.42   — тестовые сценарии и автопроверки
v0.43   — улучшение базы элементов
v0.44   — UX редактора последовательности
v0.45   — финальная структура PDF-отчёта
v0.46   — подготовка release build
```

## Что уже есть в v0.39

Итерационный диагностический цикл:

```text
форма → силы по форме → натяжения → дискретные нагрузки → новая форма → сравнение со старой → критерий сходимости
```

Основные типы:

```text
MooringIterativeSolverIteration
MooringIterativeSolverResult
MooringIterativeSolverStore
```

## Что добавлено в v0.39.1

1. Явная модель критериев:

```text
MooringIterativeSolverCriteria
```

2. Явная причина остановки solver:

```text
MooringIterativeSolverStopReason
```

3. Критерии сходимости:

```text
|ΔXсноса| ≤ OffsetToleranceM
max Δузла ≤ NodeDeltaToleranceM
|невязка Z| ≤ GeometryResidualToleranceM
```

4. Защитная остановка от грубой расходимости:

```text
|ΔX| > DivergenceOffsetChangeM
или
max Δузла > DivergenceNodeDeltaM
```

## Что добавлено в v0.39.2

1. Диагностический статус каждой итерации стал отчётным носителем критериев.

2. В существующей таблице итераций отчёта колонка `Статус` теперь содержит:

```text
reason=<StopReason>
ΔX=OK/NO
maxΔузла=OK/NO
Z=OK/NO
divergence=YES/NO
текстовую причину остановки / продолжения
```

3. Итог solver хранит и отдаёт в отчёт:

```text
FinalGeometryResidualM
Diverged
StopReason
StopReasonText
Criteria
```

4. Методическое примечание solver обновлено под v0.39.2 и прямо описывает, что отчёт по итерациям теперь показывает выполнение критериев и причину остановки.

5. В `AppInfo` отображаемая версия обновлена:

```text
v0.39.2 - iteration report diagnostics
```

## Дополнительный подготовительный сервис

В коде также есть подготовительный сервис:

```text
Services/MooringIterativeShapeComparison.cs
```

Он был добавлен как вспомогательная диагностика для будущей проверки кандидатной формы, но по плану проекта не считается основным содержанием v0.39.1/v0.39.2.

## Что не изменилось

1. Основная форма X/Z всё ещё берётся из `MooringShapeSolver`.
2. Новый solver остаётся диагностическим слоем.
3. 2D-схема не переключена на кандидатную форму.
4. PDF-схемы не переключены на кандидатную форму.
5. Дискретные нагрузки ещё не включены в основной solver — это задача v0.40.

## Как проверить

1. Обновить проект локально:

```text
Git → Pull
Build → Clean Solution
Build → Rebuild Solution
```

2. Выполнить обычный расчёт.
3. В полном отчёте найти раздел:

```text
Итерационный solver v0.39 — диагностика
```

4. В таблице итераций проверить колонку `Статус`: там должны быть `reason`, `ΔX`, `maxΔузла`, `Z`, `divergence`.
5. Проверить CI status:

```text
BuoyCalc Windows Build: success
```

## Следующий шаг

Следующий шаг по плану — v0.40: включение дискретных нагрузок в основной solver. Это нужно делать аккуратно: старую форму оставить как fallback, а новую ветку включать через отдельный переключаемый режим или внутренний кандидатный расчёт с проверкой сходимости.
