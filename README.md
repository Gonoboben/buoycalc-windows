# BuoyCalc Windows v0.38.2

Windows-ветка проекта BuoyCalc на C# + Avalonia.

## Статус

v0.38.2 уточняет поведение формы для случаев, когда длина линии меньше глубины. Волновая нагрузка не отключается и не ослабляется. Изменение касается только статусов, методических примечаний и визуального предупреждения в форме.

## Что изменилось в v0.38.2

1. Если:

```text
LineLength < Depth
```

основная форма больше не считается нормальной сходимостью поверхностной постановки.

2. `MooringShapeSolver` теперь помечает такой случай как режим:

```text
короткая линия / погружённый буй
```

3. `Converged` для основной формы становится `false`, даже если якорный узел геометрически поставлен на проектную глубину.
4. Верхний узел формы получает статус:

```text
WARNING: буй ниже поверхности из-за короткой линии
```

5. Промежуточные узлы короткой линии получают предупреждение:

```text
WARNING: участок короткой линии / погружённая постановка
```

6. Критерий сходимости в отчёте теперь явно пишет:

```text
Для поверхностной постановки требуется L ≥ Depth
```

7. Альтернативная форма с дискретными нагрузками тоже получает `WARNING`, если исходная линия короче глубины.
8. На PDF-странице сравнения форм и в 2D-схеме такой вариант должен отображаться как:

```text
альтернативная форма: WARNING
```

9. Отображаемая версия окон и `AppInfo` обновлены:

```text
v0.38.2 - short line warning
```

## Что не изменилось

Волновая нагрузка остаётся в расчёте. Даже если форма показывает погружённый буй, расчёт не отключает волну автоматически. Это сделано намеренно: нагрузка остаётся консервативной, а предупреждение относится к геометрии постановки.

## Почему это важно

В проверенных вариантах с линией 80 м и 70 м при глубине 100 м отчёт правильно давал `Не подходит`, но форма могла выглядеть слишком благополучно: якорь был на глубине, а геометрическая сходимость выглядела как `OK`. Теперь такая форма явно показывает, что это не рабочая поверхностная постановка, а погружённый режим из-за короткой линии.

## Что осталось от v0.38.1

1. Отрицательный или нулевой вес якоря в воде остаётся жёсткой ошибкой.
2. Бетонный/грузовой якорь на камне при большом запасе остаётся `Подходит с примечанием`, а не `Требуется проверка`.
3. PDF-страница сравнения форм остаётся:

```text
Сравнение форм X/Z
```


## Как проверить

1. Пересчитать вариант с линией 80 м на глубине 100 м.
2. В отчёте форма должна показывать состояние буя:

```text
под водой
```

3. Сходимость формы должна быть `нет` или `WARNING`, а не обычный `OK`.
4. Критерий должен упоминать:

```text
L ≥ Depth
```

5. Альтернативная форма должна иметь статус `WARNING`.
6. Проверить GitHub Actions:

```text
Actions -> .NET Build
```

## Следующий шаг

После проверки короткой линии и варианта с тяжёлым якорем можно продолжить сравнение PDF-вариантов. Следующий крупный инженерный шаг — итерационный solver:

```text
форма -> силы -> натяжения -> дискретные узлы -> новая форма -> сходимость
```
