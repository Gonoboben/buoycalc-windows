# BuoyCalc Windows — план архитектурной стабилизации

Дата: 2026-06-29  
Базовое состояние: `v0.46.4`  
Статус `v0.46.5`: отменён как преждевременный.

## 0. Правила этапа

Этот план описывает порядок стабилизации проекта перед новыми функциями.

На этапах 1–7 запрещено:

- менять физику расчёта;
- менять `MooringShapeSolver` по смыслу;
- добавлять 3D;
- делать точечную правку 2D как отдельную функцию;
- добавлять новые пользовательские возможности;
- маскировать архитектурную проблему очередным runtime-патчем UI.

Каждый шаг должен быть маленьким, проверяемым и обратимым.

После каждого кодового шага:

```text
1. dotnet build / CI
2. проверка BuoyCalc Windows Build: success / failure / pending
3. короткая запись в контрольные документы, если шаг меняет архитектурное состояние
```

## 0.1. Рабочее правило, чтобы не терять ход проекта

Перед каждым новым шагом нужно сверяться с этим файлом и явно называть пункт плана, который выполняется.

После каждого выполненного шага нужно обновлять `docs/CONTROL_MARK_UPDATES.md` короткой записью:

```text
что изменено;
что сознательно не трогали;
какой следующий допустимый шаг;
статус CI, если он доступен.
```

Если шаг отменён, нельзя продолжать поверх него. Сначала нужно зафиксировать отмену в `docs/CONTROL_MARK_UPDATES.md`, затем вернуться к этому плану.

## 1. Единый источник версии

### Цель

Оставить один источник актуальной версии приложения:

```text
Services/AppInfo.cs
```

`AppInfo.Version`, `AppInfo.VersionNote`, `AppInfo.WindowTitle`, `AppInfo.DisplayVersion` должны быть единственной правдой для окон, бейджей, PDF footer и пользовательских строк версии.

### Что сделать

1. Перевести `WindowVersionHelper` на использование `AppInfo`, убрать из него собственные `CurrentVersion` и `CurrentVersionNote`.
2. После этого постепенно убрать старые версии из XAML, но без массовой правки интерфейса.
3. Не добавлять новую версию ради этого шага, пока не будет отдельного решения по release policy.

### Что не делать

- Не менять solver.
- Не менять 2D-логику.
- Не менять PDF-схему.
- Не переписывать все окна сразу.

### Критерий готовности

```text
В runtime все окна показывают AppInfo.DisplayVersion или AppInfo.Version.
В проекте нет второго CurrentVersion, отличного от AppInfo.
```

## 2. Audit-скрипт против захардкоженных `v0.*` в XAML

### Цель

Не допустить возврата старых версий в пользовательский XAML.

### Что сделать

1. Добавить простой script/audit, который проверяет `.axaml` файлы на `v0.`.
2. Сначала запускать его вручную или как отдельный audit-step.
3. Затем подключить к CI, когда список исключений будет согласован.

### Область первой проверки

```text
Views/**/*.axaml
```

### Исключения

На первом шаге лучше не трогать `README.md`, `docs/*.md` и технические method notes в C# — там исторические версии могут быть осознанной документацией.

### Критерий готовности

```text
Новый XAML не может случайно принести старую v0.* версию в пользовательский интерфейс.
```

## 3. UserStatusPolicy

### Цель

Разделить технические статусы расчётного ядра и пользовательские формулировки.

Технические статусы должны остаться в solver/report diagnostics:

```text
OK / INFO / WARNING / FAILED / ERROR / Fail
```

Пользователь должен видеть понятные русские формулировки:

```text
подходит
требует проверки
не подходит
информационное примечание
форма построена
форма требует проверки
```

### Что сделать

1. Добавить `UserStatusPolicy` или аналогичный слой отображения.
2. Не менять строки статусов внутри solver и расчётного ядра.
3. Переводить статусы только на границе user-facing UI/PDF.
4. Сначала покрыть краткий итог, PDF и status summary.
5. 2D подключать только после `SelectedShapeStore`.

### Критерий готовности

```text
Технические WARNING/INFO/FAILED не попадают в пользовательский UI без явного перевода.
Полный отчёт при этом сохраняет технические статусы.
```

## 4. SelectedShapeStore

### Цель

Сделать один стабильный источник выбранной формы постановки для пользовательских выходов.

Сейчас есть несколько форм:

```text
fallback    = MooringShapeSolver / MooringShapeStore
alternative = форма с дискретными нагрузками / MooringAlternativeShapeStore
candidate   = MooringIterativeSolver.FinalShape
selected    = MooringPrimaryShapeSelector / MooringPrimaryShapeSelectionStore
```

Нужно ввести явную модель:

```text
SelectedShapeStore
SelectedShapeResult
SelectedShapeSource
SelectedShapeForUser
SelectedShapeForDiagnostics
```

Названия можно уточнить при реализации, но смысл должен быть один: пользовательские визуализации читают выбранную форму, а не сами выбирают между fallback/alternative/candidate.

### Что сделать

1. Не менять solver.
2. Не менять критерии gate.
3. Собрать read model выбранной формы после расчётного pipeline.
4. В read model сохранить:
   - координаты узлов;
   - источник формы;
   - статус для пользователя;
   - статус для диагностики;
   - наличие дискретных элементов;
   - методическое примечание для технического отчёта.
5. Пока не переподключать 2D.

### Критерий готовности

```text
Есть один объект, из которого PDF/2D/user report смогут брать форму без парсинга ReportText и без прямого выбора Store.
```

## 5. UserReportBuilder и TechnicalReportBuilder

### Цель

Перестать получать пользовательский отчёт фильтрацией одного большого технического отчёта.

Нужны два разных builder-а:

```text
UserReportBuilder
TechnicalReportBuilder
```

### UserReportBuilder

Должен строить пользовательскую модель/текст:

- краткий итог;
- основные исходные данные;
- выбранная форма;
- цепочка постановки;
- таблица элементов в пользовательских терминах;
- понятные статусы;
- короткие ограничения без solver-жаргона.

### TechnicalReportBuilder

Должен сохранить текущую глубину диагностики:

- solver tables;
- формы fallback/candidate/selected/alternative;
- gate;
- autochecks;
- vector balance;
- method notes;
- инженерные ограничения.

### Что сделать

1. Сначала выделить модель результата отчёта, не меняя текст.
2. Затем перенести полный отчёт в `TechnicalReportBuilder`.
3. Затем собрать отдельный `UserReportBuilder`.
4. После этого PDF должен читать user report model, а не `ReportText`.

### Критерий готовности

```text
PDF не парсит Markdown полного отчёта.
Полный отчёт остаётся техническим.
Пользовательский отчёт строится напрямую из расчётной модели и SelectedShapeStore.
```

## 6. Разделение user UI и technical diagnostics

### Цель

Пользовательский интерфейс должен быть понятным без знания внутренних классов.

Технические термины должны остаться только там, где окно явно техническое:

```text
Полный отчёт
diagnostics
developer/audit output
```

### Что сделать

1. Составить список user-facing строк.
2. Разделить строки на:
   - пользовательские;
   - инженерно-пояснительные;
   - технические диагностические.
3. Убрать из обычного UI:
   - `MooringShapeSolver`;
   - `fallback`;
   - `candidate`;
   - `alternative`;
   - `X/Z` без пояснения;
   - `WARNING/INFO/FAILED` без перевода;
   - исторические `v0.*`.
4. Оставить это в полном техническом отчёте.

### Критерий готовности

```text
Обычный пользовательский путь: главное окно → расчёт → PDF/2D не требует знания solver-терминов.
Технический путь: Полный отчёт сохраняет подробности.
```

## 7. Тестовый проект

### Цель

Перед следующими визуальными и физическими изменениями нужен тестовый контур.

### Что покрыть первыми тестами

1. `AppInfo` — версия и DisplayVersion не расходятся.
2. Audit XAML — нет hardcoded `v0.*` в пользовательских XAML.
3. `UserStatusPolicy` — технические статусы переводятся в пользовательские формулировки.
4. `SelectedShapeStore` — выбранная форма существует после расчёта и имеет источник.
5. Reports — PDF/user report/full report не расходятся по ключевым числам:
   - глубина;
   - длина линии;
   - снос выбранной формы;
   - количество элементов;
   - вердикт.
6. Regression для базового проекта по умолчанию.

### Что не тестировать на первом шаге

- Не пытаться сразу покрыть всю физику.
- Не создавать сложные golden PDF tests.
- Не блокировать refactor из-за отсутствия идеальной test coverage.

### Критерий готовности

```text
Есть отдельный тестовый проект, который защищает архитектурные договорённости перед правкой 2D и solver.
```

## 8. Правка 2D как отображения выбранной формы

### Почему 2D идёт до solver

2D на этом этапе не является новой физикой и не является улучшением расчёта.

Это только архитектурная очистка отображения:

```text
2D перестаёт выбирать источник формы самостоятельно
2D перестаёт парсить ReportText
2D читает готовую выбранную форму из read model
```

Если в ходе 2D-шага появляется необходимость менять расчёт, этот шаг нужно остановить и перейти к отдельному solver-плану.

### Условие входа

К этому этапу уже должны быть готовы:

- единый источник версии;
- audit против старых версий в XAML;
- UserStatusPolicy;
- SelectedShapeStore;
- разделение user/technical report;
- базовые tests.

### Что делать с 2D

1. Убрать парсинг `ReportText` из `Mooring2DCanvas`.
2. Убрать самостоятельный выбор между fallback/alternative/candidate внутри canvas.
3. Canvas должен получить готовую view/read model.
4. По умолчанию показывать одну пользовательскую выбранную форму.
5. Сравнение fallback/candidate/alternative оставить только для technical diagnostics, если оно понадобится отдельным режимом.

### Что не делать

- Не добавлять 3D одновременно.
- Не менять solver.
- Не менять критерии gate.
- Не добавлять новую форму ради визуальной красоты.

## 9. Только потом усиление физики solver

### Почему solver идёт перед 3D

Solver — инженерное ядро проекта. 3D не должен появляться раньше, чем понятно, какую форму и какую физику он показывает.

Поэтому после архитектурной стабилизации и очистки 2D следующий крупный инженерный этап — именно solver, а не 3D.

### Условие входа

К этому моменту должны быть стабильны:

- архитектура отчётов;
- пользовательские статусы;
- выбранная форма;
- 2D как отображение выбранной формы;
- тесты;
- CI.

### Возможные будущие направления

1. Уточнение нелинейного равновесия формы.
2. Более строгий учёт профиля течения по глубине.
3. Улучшение дискретных нагрузок.
4. Расширение модели якоря и грунта.
5. Валидация по Berteaux / MD&D / инженерным примерам.

### Главное правило

Физику solver нельзя усиливать одновременно с UI/refactor. Иначе невозможно понять, что именно изменило результат.

## 10. Только потом 3D

### Условие входа

3D можно обсуждать только после стабилизации выбранной формы, 2D и очередного solver-этапа.

### Правило

3D не должен придумывать физику.

Он должен читать тот же стабилизированный источник:

```text
SelectedShapeStore / SelectedShapeResult
```

или его будущую замену, если после solver-этапа будет принята новая модель выбранной формы.

### Первый допустимый 3D-шаг

Только viewer/skeleton:

- без изменения solver;
- без новых физических допущений;
- без отдельной 3D-формы;
- с явной пометкой, что это визуализация существующей расчётной формы.

## 11. Рекомендуемый первый маленький кодовый шаг после подтверждения

После утверждения этих документов первым кодовым шагом должен быть только:

```text
WindowVersionHelper читает AppInfo.Version / AppInfo.DisplayVersion,
а не собственные CurrentVersion / CurrentVersionNote.
```

Это маленький шаг, потому что:

- не меняет solver;
- не меняет 2D-логику;
- не меняет PDF-схему;
- не добавляет новых функций;
- закрывает явное архитектурное дублирование версии;
- подготавливает почву для audit-скрипта.

## 12. Предлагаемый порядок коммитов

```text
1. docs: architecture audit and refactor plan
2. docs: update refactor plan ordering and workflow rules
3. refactor: use AppInfo in WindowVersionHelper
4. audit: add XAML version scan script
5. refactor: introduce UserStatusPolicy
6. refactor: introduce SelectedShapeStore read model
7. refactor: split user and technical report builders
8. test: add BuoyCalc Windows test project
9. refactor: connect 2D to selected shape model
10. solver: physics improvements, separately planned
11. feature: 3D viewer skeleton, no solver changes
```

## 13. Stop condition

Если на любом шаге появляется желание “быстро поправить окно” или “заодно улучшить solver”, шаг нужно остановить и вернуться к этому плану.

Стабилизация считается успешной не тогда, когда добавлена новая функция, а когда:

```text
одно значение версии;
один выбранный источник формы для пользовательских выходов;
одна политика пользовательских статусов;
отдельный пользовательский отчёт;
отдельный технический отчёт;
2D/PDF/отчёт больше не расходятся по источникам данных.
```
