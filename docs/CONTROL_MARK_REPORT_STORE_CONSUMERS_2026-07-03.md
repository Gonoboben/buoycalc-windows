## Manual classification of the scan

The raw scan is intentionally simple. It reports symbol matches, not semantic consumers. Therefore this document classifies each match.

## MooringShapeStore classification

### Declaration

`MooringShapeStore` is declared in `Services/MooringShapeSolver.cs`.

Current shape-store surface:

```text
Current
Set(MooringShapeResult shape)
Clear()
```

### Real writers

#### TechnicalReportStorePublisher

`TechnicalReportStorePublisher.Publish(data)` writes `data.Shape` into `MooringShapeStore`.

This is the primary report-data publication path:

```text
TechnicalReportData -> TechnicalReportStorePublisher -> MooringShapeStore
```

#### MooringIterativeSolverStore.Set

`MooringIterativeSolverStore.Set(result)` may replace the current `MooringShapeStore` value with the selected primary shape after `MooringPrimaryShapeSelector.Select(...)` runs.

This means the iterative solver store currently has a side effect on the primary shape store:

```text
MooringIterativeSolverStore.Set(result)
  -> read fallback MooringShapeStore.Current
  -> MooringPrimaryShapeSelector.Select(fallbackShape, result)
  -> MooringPrimaryShapeSelectionStore.Set(selection)
  -> MooringShapeStore.Set(selection.Shape)
```

### Real code readers

#### MooringIterativeSolverStore.Set

`MooringIterativeSolverStore.Set(result)` reads `MooringShapeStore.Current` as the fallback shape before running primary-shape selection.

This is not a passive report consumer. It is part of the current selection/update side effect.

#### SelectedShapeStore

`SelectedShapeStore.BuildCurrent()` reads `MooringShapeStore.Current` only as a fallback read model when there is no `MooringPrimaryShapeSelectionStore.Current`.

This is the currently preferred read-model facade for consumers that need the selected shape without parsing Markdown.

### Text-only matches / not semantic consumers

These matches are not code consumers:


They are diagnostic/report strings that mention `MooringShapeStore`.

## Current architecture meaning

The current store chain is:

```text
TechnicalReportData
  -> TechnicalReportStorePublisher.Publish(data)
     -> MooringShapeStore.Set(data.Shape)
     -> MooringIterativeSolverStore.Set(data.IterativeSolver)
        -> MooringPrimaryShapeSelector.Select(...)
        -> MooringPrimaryShapeSelectionStore.Set(selection)
        -> MooringShapeStore.Set(selection.Shape)
```

This means `MooringShapeStore` may first receive the report shape and then receive the gate-selected primary shape.

`SelectedShapeStore` is already acting as the shape-facing read-model facade:

```text
MooringPrimaryShapeSelectionStore.Current
  -> SelectedShapeStore.Current
     -> fallback MooringShapeStore.Current only if gate selection is absent
```

## Boundary decision

Future PDF / 2D / UI / report consumers should move toward explicit read-model data such as `SelectedShapeStore` or a successor read model.

They should not:

- parse Markdown to recover engineering state;
- infer primary shape from presentation text;
- read solver internals directly;
- introduce visual-only shape selection logic.
