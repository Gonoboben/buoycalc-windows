## Manual classification of the scan

The scan intentionally reports symbol matches, not semantic consumers. This section classifies each relevant match.

## Actual selected-shape consumers

### PDF consumer

`Services/PdfReportBuilder.cs` reads:

```text
SelectedShapeStore.Current
```

Classification:

```text
PDF renderer / PDF diagram source selection
```

Meaning:

- PDF already reads the selected-shape read model.
- PDF still has fallback metric parsing in the same selection function for cases where no selected shape exists.
- This fallback parsing should not be expanded.
- Future cleanup should prefer explicit read-model data over report-text parsing.

### 2D consumer

`Views/Mooring2DCanvas.cs` reads:

```text
SelectedShapeStore.Current
```

Classification:

```text
2D renderer / engineering shape drawing
```

Meaning:

- 2D already reads the selected-shape read model.
- 2D uses selected shape when at least two nodes are available.
- 2D also checks the alternative-shape store for comparison drawing.
- Future cleanup should keep 2D drawing based on explicit read-model data, not Markdown text.

## SelectedShapeStore role

`Services/SelectedShapeStore.cs` is currently the selected-shape read-model facade.

Current behavior:

```text
MooringPrimaryShapeSelectionStore.Current
  -> SelectedShapeStore.Current
     -> fallback MooringShapeStore.Current only if gate selection is absent
```

This means `SelectedShapeStore` prefers gate-selected primary shape data and falls back to the current shape store only during architecture stabilization.

They are useful for audit context, but should not be treated as proof of renderer ownership.

## Current architecture meaning

Current selected-shape flow:

```text
MooringIterativeSolverStore.Set(result)
  -> MooringPrimaryShapeSelector.Select(fallbackShape, result)
  -> MooringPrimaryShapeSelectionStore.Set(selection)
  -> SelectedShapeStore.Current
     -> PDF renderer
     -> 2D renderer
```

Fallback flow:

```text
MooringShapeStore.Current
  -> SelectedShapeStore.Current
     -> PDF renderer / 2D renderer only when no gate selection exists
```

## Boundary decision

PDF and 2D are already the first consumers using `SelectedShapeStore`.

The next cleanup should not move them away from `SelectedShapeStore`. Instead, it should reduce remaining fallback/report-text parsing and make the explicit read-model boundary clearer.

## Recommended next cleanup order

1. Document the PDF fallback path in `PdfReportBuilder.SelectDiagramSource(...)`.
2. Decide whether PDF should receive explicit selected-shape data instead of reading global stores.
3. Document the 2D selected-shape path in `Mooring2DCanvas`.
4. Decide whether 2D should receive an explicit view-model/read-model input instead of reading global stores.
5. Only after those decisions, make one small behavior-preserving refactor at a time.
