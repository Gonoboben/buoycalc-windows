# BuoyCalc — контрольная отметка проекта

Дата начальной отметки: 2026-06-29  
Репозиторий: `Gonoboben/buoycalc-windows`  
Текущая логическая версия: `v0.46.4`  
Назначение: постоянный контрольный документ проекта. Его нужно обновлять после каждого значимого шага, чтобы не потерять инженерную правду, текущие ограничения и порядок дальнейших работ.

---

## 1. Главный статус

BuoyCalc сейчас — инженерный прототип / pre-engineering tool для предварительной компоновки буя, линии, соединителей, приборов и якоря.

BuoyCalc пока нельзя считать полноценной профессиональной программой для окончательного расчёта океанографической постановки или швартовки катера/катамарана.

Главная причина: физическое ядро пока не доказано как полный равновесный solver. Не хватает формального force residual, согласованности формы и сил, seabed contact, mode-specific wave physics и benchmark/validation against reference solver.

Правильная формулировка текущего доверия:

```text
BuoyCalc можно использовать для предварительной инженерной оценки и выявления проблем,
но нельзя использовать как единственный расчёт для реальной постановки в море.
```

---

## 2. Принципы, которые нельзя нарушать

1. PDF, 2D, будущая 3D-схема и отчёты не должны придумывать физику.
2. Любая координата должна приходить из расчётной модели / solver, а не из визуального костыля.
3. В пользовательском PDF должна быть только одна выбранная расчётная форма.
4. Внутренние формы, fallback, gate, diagnostic rows и solver history должны жить в полном техническом отчёте, а не в пользовательском PDF.
5. Английские служебные строки допустимы только как внутренние имена классов/полей в полном техническом отчёте, но не в пользовательском PDF.
6. `L < Depth` не является ошибкой само по себе. Это может быть нормальный submerged/short-line режим, если пользователь именно его задал.
7. Форма с дискретными элементами физически ближе к натурной цепочке, чем упрощённая fallback-форма.
8. Gate/fallback должны отвечать на вопрос численной надёжности, а не на вопрос физической реалистичности.
9. Вердикт `Подходит` допустим только с честной оговоркой о классе модели и предупреждениях solver.
10. Не путать задачи: океанографический буй и швартовка катера/катамарана — разные классы задач.

## 4. Что пока НЕ готово

### 4.1 Физика solver

Пока не доказано:

1. Полное статическое равновесие всей системы.
2. Явный residual по сумме сил.
3. Согласованность углов формы с углами из сил.
4. Реальный seabed contact / touchdown.
5. Корректное поведение тяжёлой нижней цепи на грунте.
6. Реакции якоря и грунта.
7. Валидированная wave physics для surface и submerged режимов.
8. Сравнение с эталонным solver вроде MoorDyn.
9. Валидация на натурных или лабораторных данных.

### 4.2 Якорь

Текущая модель якоря — предварительная.

Проблемы:

```text
нет полноценной реакции грунта
нет угла тяги на якорь как физического условия
нет uplift/contact logic
нет циклической нагрузки
нет проверки типа грунта на уровне инженерного стандарта
```

Вывод: запас якоря в текущем BuoyCalc можно читать только как предварительный индикатор.

### 4.3 Волна

Нужно разделить:

```text
surface buoy wave load
submerged buoy wave load with depth decay
```

Для погружённого буя нельзя просто применять поверхностную волновую нагрузку без модели затухания орбитальной кинематики с глубиной.

### 4.4 Швартовка катера/катамарана

Текущий BuoyCalc не является полноценным инструментом для швартовки катера.

Для такой задачи дополнительно нужны:

```text
6DOF корпус
ветер
волна
течение
yaw/weathervaning
bridle geometry
точки крепления на корпусе
рывки и динамика
chafe
запасы по кнехтам/уткам/швартовым элементам
```

## 5. Текущие известные проблемы UI/отчётов

1. 2D-окно ещё требует отдельной правки: там исторически смешаны исходная и форма с дискретными элементами.
2. В 2D-схеме могут оставаться старые слова `основная`, `альтернативная`, `WARNING`.
3. 3D-окно пока не добавлено.
4. User PDF и full report надо проверять после каждой правки, чтобы не вернулись fallback/candidate-конфликты.

Пользовательские уровни статуса, к которым нужно прийти:

```text
ОК
Требует проверки
Расчёт ограничен
Ошибка исходных данных
```

## 7. Что считать рабочим состоянием

### Минимально рабочее состояние

BuoyCalc можно считать рабочим pre-engineering инструментом, когда:

1. Пользовательский PDF короткий и понятный.
2. Полный отчёт отделён и содержит всю диагностику.
3. В user PDF одна выбранная форма и одно значение сноса.
4. Нет необъяснённого Warning.
5. Режим surface/submerged трактуется корректно.
6. Есть trust checks.
7. Программа честно показывает ограничения модели.

### Инженерно доверенное состояние

BuoyCalc можно считать инженерно доверенным для oceanographic buoy предварительно-проектного уровня, когда дополнительно:

1. Есть force residual.
2. Есть angle consistency.
3. Есть seabed/contact logic.
4. Есть mode-specific wave physics.
5. Есть benchmark against reference solver.
6. Есть regression suite.
7. Есть хотя бы один validation case по натурным/лабораторным данным.

Использовать:

```text
пользовательский PDF-отчёт
полный технический отчёт
форма с дискретными элементами
выбранная расчётная форма
расчётная 2D-схема
расчётная 3D-схема
предварительная инженерная оценка
ограничение модели
требует технической проверки
```

Не использовать в пользовательском PDF:

```text
fallback
candidate
primaryShape
alternative shape
основная форма
альтернативная форма
MooringShapeSolver fallback
GeometryNotClosed без пояснения
WARNING без пояснения
```

## 9. Ошибки, которые нельзя повторять

1. Называть намеренный `L < Depth` ошибкой, если это submerged case.
2. Показывать в PDF два разных сноса с одинаковым названием.
3. Выводить fallback-форму в пользовательский PDF.
4. Писать, что форма плохая только потому, что gate её не сделал основной.
5. Смешивать физическую реалистичность и численный допуск gate.
6. Скрывать ограничения модели за словом `Подходит`.
7. Делать 3D-схему как декоративную картинку без связи с расчётными узлами.
8. Объявлять программу готовой к морской постановке без benchmark и validation.

## 10. Правило обновления этого файла

После каждого значимого шага нужно обновлять этот документ:

1. Добавить новую версию в раздел `Что уже сделано`.
2. Перенести выполненные пункты из плана в статус `сделано`.
3. Обновить текущие известные проблемы.
4. Обновить следующий шаг.
5. Не удалять ограничения модели только потому, что интерфейс стал выглядеть лучше.
