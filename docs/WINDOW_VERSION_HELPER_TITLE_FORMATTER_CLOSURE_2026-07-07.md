# WindowVersionHelper title formatter closure

Date: 2026-07-07
Related issue: #154
Scope: architecture decision / documentation only

## Decision

Retain `WindowVersionHelper` as the common title formatter.

After issue #151, the helper has one responsibility and no delayed mutation:

```csharp
public static void Apply(Window window, string titlePrefix)
{
    window.Title = titlePrefix + " " + AppInfo.Version;
}
```

This is an appropriate shared presentation boundary because four active windows use the same exact formatting rule with different prefixes.

No production change is justified by this audit.

## Current application metadata

`AppInfo` currently provides:

```text
Version = v0.46.4
VersionNote = пользовательские статусы PDF
WindowTitle = BuoyCalc Windows v0.46.4
DisplayVersion = v0.46.4 - пользовательские статусы PDF
```

`WindowVersionHelper` depends only on `AppInfo.Version`.

## Active callers

The helper is called immediately after `AvaloniaXamlLoader.Load(this)` by:

```text
MainWindow
CurrentProfileWindow
Mooring2DWindow
ReportTextWindow
```

For every caller, title assignment is synchronous during construction and before the owned window is shown.

## Exact prefixes and final titles

```text
MainWindow
prefix: BuoyCalc Windows
final:  BuoyCalc Windows v0.46.4

CurrentProfileWindow
prefix: Профиль течения по глубине
final:  Профиль течения по глубине v0.46.4

Mooring2DWindow
prefix: 2D-схема постановки
final:  2D-схема постановки v0.46.4

ReportTextWindow
prefix: Полный текстовый отчёт
final:  Полный текстовый отчёт v0.46.4
```

The report window intentionally has different strings for different presentation roles:

```text
window title prefix: Полный текстовый отчёт
visible page header: Полный отчёт
badge: технический отчёт
```

The helper must not infer a title from the page header or badge.

## MainWindow initial XAML title

`MainWindow.axaml` currently initializes `Title` from `AppInfo.WindowTitle` during XAML load. The constructor then assigns the same final value through `WindowVersionHelper.Apply(...)`.

This duplicate initial assignment is harmless:

- both sources currently produce the exact same final string;
- the helper assignment remains the common constructor-time boundary used by all four callers;
- removing only the MainWindow helper call would create a special case;
- changing only the XAML title back to a base prefix would not improve runtime behavior;
- removing `AppInfo.WindowTitle` requires a separate usage audit and provides no user-facing value here.

Therefore no MainWindow production change is included.

## Removal alternatives evaluated

### Duplicate code-behind concatenation

Replacing the helper with this statement in four constructors:

```csharp
Title = prefix + " " + AppInfo.Version;
```

would remove one small file while duplicating the formatting rule and version dependency four times.

Decision: reject.

### Full-title property per window

Adding properties such as:

```text
AppInfo.CurrentProfileWindowTitle
AppInfo.Mooring2DWindowTitle
AppInfo.ReportTextWindowTitle
```

would move window-specific Russian presentation copy into a global metadata class and increase API surface solely to delete one formatter.

Decision: reject.

### XAML formatting or bindings

Constructing each title from a prefix and `AppInfo.Version` in XAML would require repeated markup, formatting bindings, or additional resources. It would be more complex than the current one-line constructor call and would need separate XAML compiler validation.

Decision: reject.

### Hard-coded full titles

Writing `v0.46.4` into each XAML file would reintroduce stale version literals and require coordinated edits on every version bump.

Decision: reject.

## Retained boundary

```text
AppInfo owns application version metadata.
WindowVersionHelper owns the exact shared window-title composition rule.
Each window owns its title prefix and constructor call.
XAML owns visible page headers and badges.
ViewModels do not own application-version presentation.
```

## Future change criteria

Reconsider the helper only when one of these becomes true:

- all window titles are generated by an existing shared XAML resource mechanism without repeated formatting markup;
- application metadata exposes a deliberate title model for all windows for another concrete requirement;
- the application adopts a window shell/navigation architecture that already owns title composition;
- only one caller remains.

Do not remove the helper solely to reduce file or method count.

## Output invariants

This audit changes no production behavior. The following remain unchanged:

```text
- all four final window titles
- constructor order
- XAML headers and badges
- AppInfo values
- window ownership and modality
- commands and dialogs
- current profile behavior
- library behavior
- report and PDF behavior
- 2D rendering
- project storage, JSON and DTOs
- calculation core, solver and engineering physics
```

No 3D is introduced.
