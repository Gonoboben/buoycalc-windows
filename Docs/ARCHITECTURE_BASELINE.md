# Архитектурная база проекта

Дата фиксации: 2026-06-30

Документ фиксирует базовую архитектуру, от которой должны отталкиваться дальнейшие изменения. Он не заменяет README, а задаёт инженерные ограничения, которые нельзя нарушать без отдельного архитектурного решения.

## 1. BuoyCalc Windows — текущая база

Репозиторий: `Gonoboben/buoycalc-windows`

Текущая видимая версия README на момент фиксации: `v0.46.4`.

### 1.1. Назначение

Windows-ветка BuoyCalc — инженерное приложение для расчёта морской постановки, где пользователь собирает не условные «участки», а физическую последовательность элементов.

Базовая логика:

```text
буй → соединители → линии → приборы → цепь → якорь
```

Приборы, скобы, вертлюги, соединители и другие дискретные элементы считаются частью последовательности, а не внешними примечаниями.

### 1.2. Расчётное ядро

Базовые расчётные блоки:

1. плавучесть;
2. вес в воде;
3. действие течения;
4. действие волны;
5. слабое звено;
6. запас якоря;
7. сегментированная линия;
8. основная X/Z-форма;
9. альтернативная X/Z-форма;
10. shape-based силы;
11. shape-based натяжения;
12. позиционная модель последовательности по `s`;
13. дискретные элементы как нагрузки вдоль формы;
14. 2D-схема;
15. PDF-отчёт.

### 1.3. Solver policy

Старый solver не удаляется до полной проверки нового.

Новый итерационный solver должен существовать отдельно:

```text
старый solver → остаётся рабочей базой
новый solver → отдельный сервис/результат/диагностика
```

До верификации нового solver:

1. не удалять старый solver;
2. не удалять gate;
3. не удалять fallback-форму;
4. не подменять инженерный результат косметическим выводом;
5. не считать warning ошибкой без анализа причины.

### 1.4. Визуализация и PDF

PDF и 2D должны отображать расчётную модель, а не декоративную схему.

Зафиксированные ограничения:

1. X/Z должны быть инженерно согласованы.
2. Равный масштаб по X и Z предпочтителен для физического восприятия формы.
3. PDF-схема должна брать координаты из расчётного слоя дискретных нагрузок.
4. Пользовательский PDF может быть упрощён, но не должен скрывать инженерные предупреждения.
5. Технические предупреждения должны переводиться в понятные пользовательские формулировки без потери смысла.

### 1.5. Ближайший шаг по README

Следующий шаг после `v0.46.4`:

```text
v0.46.5 — перевести 2D-окно на ту же пользовательскую терминологию.
```

Физика solver при этом не должна изменяться.

## 2. PDU / энергетика БНПА «Видеомодуль» — текущая база

Репозиторий-кандидат: `Gonoboben/plataVM`

Активная инженерная ветка проекта ранее фиксировалась как плата питания БНПА «Видеомодуль»:

```text
PDU-BNPA-PWR-V1.1 / V1.2
```

### 2.1. Энергетическая база

Базовые параметры:

1. аккумуляторная химия: `LiFePO4`;
2. номинальное напряжение: `12,8 В`;
3. суммарная ёмкость: `288 А·ч`;
4. осветители: `6 шт × 30 Вт = 180 Вт`;
5. прочая электроника: приблизительно `40 Вт`;
6. суммарная полезная нагрузка: приблизительно `220 Вт`.

### 2.2. Зафиксированная архитектура V1.2

Базовое решение:

```text
BAT1 + BAT2 → параллельно → BAT_BUS_4S
```

Отдельный аварийный блок:

```text
EMG_4S2P → только CRITICAL_INPUT / 5V_CRIT / 3V3_CRIT
```

Ограничение:

```text
EMG_4S2P не питает ИЗС и BOOST 32 В
```

Основной интерфейс с главным модулем:

```text
RS-485
```

Сервисные интерфейсы:

```text
UART / SWD
```

### 2.3. Запреты для PDU-линии

1. Не смешивать аварийное питание с силовым питанием осветителей.
2. Не объединять Power GND и Signal GND без STAR POINT/обоснованной схемы земли.
3. Не менять архитектуру BAT_BUS_4S без расчёта последствий.
4. Не переносить решения из BuoyCalc в PDU и наоборот без отдельного ADR.

## 3. Правило изменения этой базы

Этот файл можно менять только отдельным PR с явным описанием:

1. какая часть архитектуры меняется;
2. почему прежняя архитектура недостаточна;
3. какие риски появляются;
4. какие проверки должны подтвердить корректность;
5. какие прежние решения сохраняются.
