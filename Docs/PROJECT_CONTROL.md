# Управление проектом BuoyCalc Windows

Дата фиксации: 2026-06-30

Этот документ задаёт правила ведения **только проекта BuoyCalc Windows** в GitHub. Его цель — не дать проекту потерять архитектурную логику при последующих доработках, особенно при работе через ИИ-ассистента.

Важно: `BuoyCalc Windows` и `PlataVM / плата питания` — это абсолютно разные проекты. У них разные репозитории, разные архитектуры, разные хронологии и разные контрольные документы.

## 1. Назначение

`BuoyCalc Windows` ведётся как инженерная расчётная система с фиксируемой хронологией, архитектурными решениями, проверяемыми изменениями и воспроизводимыми версиями.

Каждое изменение должно отвечать на вопросы:

1. Что изменено?
2. Почему это изменено?
3. Какая часть архитектуры BuoyCalc Windows затронута?
4. Не нарушены ли прежние зафиксированные решения BuoyCalc Windows?
5. Как проверено, что расчётная логика не деградировала?

## 2. Граница проекта

В этом репозитории ведётся только:

```text
BuoyCalc Windows / расчёт буя / расчёт морской постановки
```

В состав проекта входят:

1. модель последовательности элементов постановки;
2. расчёт плавучести, веса в воде, течения, волны и натяжений;
3. solver и fallback-логика;
4. 2D-визуализация;
5. PDF/текстовый отчёт;
6. библиотека элементов;
7. пользовательские проекты и настройки Windows-приложения.

Не входят в этот проект:

1. `PlataVM`;
2. плата питания БНПА;
3. PDU;
4. схемотехника питания;
5. энергетика БНПА «Видеомодуль»;
6. прошивка платы питания;
7. аппаратная архитектура аккумуляторного комплекса.

Эти направления должны вестись в отдельном репозитории и отдельной хронологии.

## 3. Базовое правило архитектуры

Архитектура BuoyCalc Windows не меняется «по ходу дела».

Любое изменение, которое затрагивает расчётное ядро, структуру данных, solver, модель последовательности, 2D-визуализацию или отчётную логику, должно сопровождаться записью в:

- `Docs/ARCHITECTURE_BASELINE.md` — если фиксируется новая базовая архитектура;
- `Docs/CHRONOLOGY.md` — если это шаг реализации;
- ADR-документе `Docs/adr/ADR-YYYY-MM-DD-*.md` — если это архитектурное решение.

## 4. Запрет на скрытые изменения

Запрещено:

1. Заменять старый solver новым без параллельной проверки.
2. Удалять fallback-форму до полной верификации нового solver.
3. Менять физику расчёта одновременно с косметикой интерфейса/отчёта.
4. Менять последовательную модель элементов на жёсткие «участки 1/2/3».
5. Переписывать отчёт ради внешнего вида, если это ухудшает инженерную трассируемость.
6. Переносить в BuoyCalc Windows решения из PlataVM/PDU/платы питания.
7. Хранить в хронологии BuoyCalc Windows архитектурные решения PlataVM.

## 5. Правило работы через GitHub

Работа ведётся через ветки и pull request.

Минимальный цикл изменения:

1. Создать ветку под задачу.
2. Внести минимальный инженерно связанный набор изменений.
3. Обновить хронологию BuoyCalc Windows.
4. Проверить сборку и CI.
5. Описать в PR:
   - цель;
   - затронутые файлы;
   - архитектурный риск;
   - проверку;
   - что специально не изменялось.

## 6. Роль ИИ-ассистента

Перед предложением или внесением изменений по BuoyCalc Windows ассистент должен свериться с:

1. текущим README;
2. `Docs/PROJECT_CONTROL.md`;
3. `Docs/ARCHITECTURE_BASELINE.md`;
4. `Docs/CHRONOLOGY.md`;
5. открытыми issues/PR;
6. последними коммитами и CI-статусом.

Ассистент не должен уводить BuoyCalc Windows в новую архитектуру без отдельного ADR и явного объяснения последствий.

Если задача относится к PlataVM, PDU или плате питания, ассистент должен остановить работу в репозитории BuoyCalc Windows и перейти к отдельному репозиторию/отдельной хронологии этого проекта.
