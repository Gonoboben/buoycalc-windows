# Управление проектом «Архитектор инженерии»

Дата фиксации: 2026-06-30

Этот документ задаёт правила ведения инженерного проекта в GitHub. Его цель — не дать проекту потерять архитектурную логику при последующих доработках, особенно при работе через ИИ-ассистента.

## 1. Назначение

Проект ведётся как инженерная система с фиксируемой хронологией, архитектурными решениями, проверяемыми изменениями и воспроизводимыми версиями.

Каждое изменение должно отвечать на вопросы:

1. Что изменено?
2. Почему это изменено?
3. Какая часть архитектуры затронута?
4. Не нарушены ли прежние зафиксированные решения?
5. Как проверено, что расчётная/инженерная логика не деградировала?

## 2. Активные линии проекта

На дату фиксации выделены отдельные линии, которые нельзя смешивать без явного архитектурного решения:

1. `BuoyCalc Windows / расчёт буя` — приложение инженерного расчёта постановки: буй, соединители, линии, приборы, цепь, якорь, расчёт формы, нагрузок, натяжений, 2D/PDF-отчёт.
2. `BuoyCalc iOS` — отдельная ветка/реализация, не должна автоматически диктовать архитектуру Windows-версии.
3. `PDU / энергетика БНПА «Видеомодуль»` — отдельная инженерная линия по питанию и плате управления энергией.

## 3. Базовое правило архитектуры

Архитектура не меняется «по ходу дела».

Любое изменение, которое затрагивает расчётное ядро, структуру данных, solver, модель последовательности, отчётную логику или энергетическую архитектуру, должно сопровождаться записью в:

- `Docs/ARCHITECTURE_BASELINE.md` — если фиксируется новая базовая архитектура;
- `Docs/CHRONOLOGY.md` — если это шаг реализации;
- ADR-документе `Docs/adr/ADR-YYYY-MM-DD-*.md` — если это архитектурное решение.

## 4. Запрет на скрытые изменения

Запрещено:

1. Заменять старый solver новым без параллельной проверки.
2. Удалять fallback-форму до полной верификации нового solver.
3. Менять физику расчёта одновременно с косметикой интерфейса/отчёта.
4. Менять последовательную модель элементов на жёсткие «участки 1/2/3».
5. Смешивать Windows, iOS и PDU-архитектуры без отдельного решения.
6. Переписывать отчёт ради внешнего вида, если это ухудшает инженерную трассируемость.

## 5. Правило работы через GitHub

Работа ведётся через ветки и pull request.

Минимальный цикл изменения:

1. Создать ветку под задачу.
2. Внести минимальный инженерно связанный набор изменений.
3. Обновить хронологию.
4. Проверить сборку и CI.
5. Описать в PR:
   - цель;
   - затронутые файлы;
   - архитектурный риск;
   - проверку;
   - что специально не изменялось.

## 6. Роль ИИ-ассистента

Перед предложением или внесением изменений ассистент должен свериться с:

1. текущим README;
2. `Docs/PROJECT_CONTROL.md`;
3. `Docs/ARCHITECTURE_BASELINE.md`;
4. `Docs/CHRONOLOGY.md`;
5. открытыми issues/PR;
6. последними коммитами и CI-статусом.

Ассистент не должен уводить проект в новую архитектуру без отдельного ADR и явного объяснения последствий.
