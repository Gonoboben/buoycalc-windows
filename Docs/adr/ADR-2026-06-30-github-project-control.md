# ADR-2026-06-30: Ведение проекта через GitHub с архитектурной хронологией

Статус: принято к рассмотрению через PR

Дата: 2026-06-30

## Контекст

Проект развивается как инженерная система. В нём уже есть несколько направлений:

1. BuoyCalc Windows / расчёт буя.
2. BuoyCalc iOS.
3. PDU / энергетика БНПА «Видеомодуль».

При продолжительной работе возникает риск, что новые изменения начнут незаметно менять архитектуру: solver, структуру последовательности элементов, расчётную модель, PDF/2D-представление или энергетическую схему.

Такой дрейф опасен: устройство или расчётная система могут стать нестабильными, а результаты — недостоверными.

## Решение

Вести проект через GitHub с обязательными контрольными документами:

1. `Docs/PROJECT_CONTROL.md` — правила ведения проекта.
2. `Docs/ARCHITECTURE_BASELINE.md` — зафиксированная архитектурная база.
3. `Docs/CHRONOLOGY.md` — последовательная хронология изменений.
4. `Docs/adr/` — каталог архитектурных решений.

Все изменения, влияющие на архитектуру, выполняются только через отдельную ветку и PR.

## Последствия

Положительные:

1. Появляется единая точка сверки перед каждым новым шагом.
2. Снижается риск случайной смены архитектуры.
3. Можно восстановить логику: почему было принято то или иное решение.
4. Можно отделять расчётные изменения от косметических.
5. Можно вести несколько направлений проекта без смешивания архитектур.

Ограничения:

1. Документацию нужно обновлять после каждого значимого изменения.
2. Нельзя быстро «переделать как кажется лучше», если это ломает зафиксированную модель.
3. Новые идеи сначала проходят через ADR или отдельный план внедрения.

## Проверка

Для каждого будущего PR проверяется:

1. обновлена ли хронология;
2. не нарушена ли архитектурная база;
3. не смешаны ли независимые линии проекта;
4. не изменилась ли физика расчёта без явного указания;
5. пройден ли CI, если затронут код.
